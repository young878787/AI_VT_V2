# AI_VT_V2 MVP 缺口與優先路線圖

更新日期：2026-04-14

## 0. MVP 定義（先定義才不會失焦）

建議你把目前 MVP 明確定義為：

「一個能穩定陪聊、具情緒表達、可記住使用者、可連續使用的 AI VTuber。」

關鍵詞是：穩定、連續、可感知價值。
不是功能越多越好。

---

## 1. 優先級總覽

### P0（必做，沒有就不算 MVP）

1. 啟動與連線健康檢查流程
2. 對話會話管理與恢復體驗
3. 記憶寫入品質控管
4. 錯誤處理與可觀測性
5. 成本與安全護欄

### P1（強化實用性，做了體驗差很多）

1. 語音輸入（STT）形成完整語音對話
2. Persona / 情緒模式切換
3. 快捷操作與對話效率功能
4. 直播模式 UI（精簡控制、清爽展示）

### P2（中長期增長功能）

1. 多角色協作與場景腳本
2. 長期記憶升級（檢索式 / 主題式）
3. 外部工具整合（行程、提醒、資訊查詢）
4. 發布與部署標準化

---

## 2. P0 詳細說明

## P0-1 啟動與連線健康檢查流程

現況：
- 前端能提示後端是否連線，但仍偏被動。

缺口：
- 使用者無法快速知道「哪一段壞掉」（API key、provider、TTS、WebSocket、模型資源）。

完成標準：
- 一個健康檢查面板，啟動後 5 秒內明確顯示各模組狀態。

---

## P0-2 對話會話管理與恢復體驗

現況：
- 已有可選 session_id 持久化。

缺口：
- 使用者體感上仍像「斷線就可能失去上下文」。

完成標準：
- 可見的會話狀態（當前 session、最後保存時間）
- 重整頁面可恢復最近對話
- 可手動開新會話 / 清除當前會話

---

## P0-3 記憶寫入品質控管

現況：
- 已有 profile 與 memory append。

缺口：
- 容易重複、泛化或留下低價值記錄。

完成標準：
- 記憶寫入前做去重與重要性分級
- 將短期事件與長期偏好分離
- 提供簡易記憶檢視與手動修正入口

---

## P0-4 錯誤處理與可觀測性

現況：
- 有 console log 與基本錯誤訊息。

缺口：
- 缺乏事件級追蹤（一次對話到底經過哪些步驟）。

完成標準：
- 每輪對話有 trace_id
- 後端分段記錄：收訊息、LLM、tool call、stream、TTS
- 前端可顯示「目前在哪個階段」

---

## P0-5 成本與安全護欄

現況：
- 已有 token 壓縮閾值與 provider 檢查。

缺口：
- 缺少每輪成本預估、節流策略、濫用防護。

完成標準：
- 每輪 token 使用量與估算成本可記錄
- 超頻訊息頻率時限流
- 基本敏感內容策略（至少記錄與告警）

---

## 3. P1 詳細說明（實用性提升）

## P1-1 語音輸入（STT）

價值：
- 真正接近「VTuber 對話」使用情境。

建議：
- 可先做瀏覽器 Web Speech API 原型，再評估雲端 STT 品質。

---

## P1-2 Persona / 情緒模式切換

價值：
- 同一角色可有不同陪伴風格（例如：活潑、治癒、吐槽）。

建議：
- 先做 3 種模式，切 system prompt 片段與表情偏好範本。

---

## P1-3 快捷操作

價值：
- 提高頻繁使用者效率。

建議：
- 重送上一句、要求更短回覆、切換「只聊天 / 聊天+動作 / 聊天+語音」。

---

## 4. 這一週最小路線（建議）

Day 1-2：
- 做 P0-1 健康檢查 + P0-2 會話狀態可視化

Day 3-4：
- 做 P0-4 trace_id 與對話階段可視化

Day 5-7：
- 做 P0-3 記憶去重 + P0-5 基本限流與成本記錄

完成後你會得到：
- 一個更像「可每天使用」而不是「技術展示」的 VTuber 系統。
